Glossary

Bracket code
A short id for a match slot, like M74. The tree is a map of codes.
Topology
The fixed shape of the knockout tree. Which two prior matches feed each match.
Derivation
Every pixel comes from the data. Nothing about the result is hardcoded.
Pure function
Same inputs give the same output, with no side effects.
Wire format
The rule for packing a bracket into a URL. base64url of a small JSON object.
URL fragment
The part of a link after the number sign. Browsers never send it to a server.
Hermetic test
A test using only frozen inputs, never live data.

The evolution is the spine

Every refactor fixed the pain of the last one. Hold this arc in mind.

  1. Hardcoded HTML. One file, everything inline. Hard to change, impossible to keep current.
  2. JSON data layer. Pull picks and results out of code into data/*.json. Content and presentation split.
  3. Python generator. build_dashboard.py derives the whole page from the data.
  4. Self-updating machine. A GitHub Action syncs results and redeploys on a schedule.
  5. Client-side pure function. Port the generator to the browser so anyone drops in a file.
  6. URL-encoded state. Pack the bracket into a link, so sharing needs no server.
  7. Zero-backend social. A local leaderboard compares links, and privacy holds by physics.
The running example

One real match from the demo bracket. M74, Round of 32. Germany 1, Paraguay 1, Paraguay win 4 to 3 on penalties. You picked Paraguay to advance. You will watch this one pick move from a spreadsheet cell, to a derived bracket slot, to a scored result, to a single bit inside a link.

01
The spark

From a spreadsheet to a spec

Lead voice: Sean Grove. State the intent before the code.

A leader in my org ran a bracket challenge. He emailed an Excel workbook. Everyone filled in 31 knockout picks, a champion, and a tiebreaker, and emailed it back. The data wanted to be visual, so I designed a dashboard in Microsoft Cowork. A glassmorphism board, KPI cards, and a bracket map.

The move was not the design. Instead of keeping it as a one-off prompt, I wrote the design down as a markdown build kit. A spec anyone hands to an AI to rebuild the same dashboard with their own picks. Version 6, then 7, then 8, each one tighter.

ELI5

I did not paint one picture. I wrote the recipe, so anyone bakes the same cake with their own ingredients.

Technical

The spec fixed the data contract before any repo existed. A bracket is an entrant, a tiebreaker, 31 round picks, a champion, and a runner-up. Every later stage, Python generator, browser port, and URL encoder, reads that same shape. The contract stayed stable while the runtime changed three times.

The tradeoff

Spec first costs an hour and saves the project. A prompt makes one dashboard. A written spec makes a data contract every runtime agrees on. The spec is the product. The code is the output.

Write the recipe, not the dish.

02
Make it mine

The generator and the machine

wc26-bracket Python + GitHub Actions

Lead voices: Karpathy, then Russinovich. Build the simplest generator, then show what the bot does underneath.

The first repo turned one hardcoded HTML file into a real architecture. Three JSON files hold the truth. A Python script derives the page. A bot keeps it current. The whole thing serves as static files on GitHub Pages, with no server and no build step.

The data model

Three files, one contract. Content lives in data, not in code.

Technical

The generator loads the three files at the top. Tuples are restored from JSON lists so the render code reads cleanly.

wc26-bracket/scripts/build_dashboard.py:18-25
P=_load("picks.json"); L=_load("live.json"); T=_load("topology.json")
ENTRANT=P["entrant"]; TIEBREAKER=P["tiebreaker"]
R32=[tuple(r) for r in P["r32"]]
RES={c:tuple(v) for c,v in L["res"].items()}
KO_FEED={c:tuple(v) for c,v in T["ko_feed"].items()}

picks.json holds your bracket. live.json holds results the bot writes, as res {code:[gA,gB,winner,note]}. topology.json holds ko_feed, the map from a match to its two feeder matches.

The render engine derives everything

ELI5

The page is not written by hand. The script reads your picks and the scores, then works out every cell, card, and sentence from that data. Change the data, and the page changes with it.

Technical

Later-round match codes are derived from feeder codes, not stored. Story cards and the stage tracker are computed the same way, so a rebuild always matches the data.

wc26-bracket/scripts/build_dashboard.py:54-73
def _derive_round(prev_codes, wins, short):
    codes=[]
    for j,w in enumerate(wins):
        c=_code_for(prev_codes[2*j], prev_codes[2*j+1])
        PICK_BY_CODE[c]=w; KO_ROUND[c]=short; codes.append(c)
    return codes

The sync engine, and one messy edge

ELI5

A small bot checks for finished games a few times a day, writes the scores into one file, and never erases a good result. One game type lies about its score, so the bot corrects it.

Technical

The sync matches finished games to bracket codes round by round, walking ko_feed so a match resolves only once both feeders have a winner.

wc26-bracket/scripts/fetch_results.py, match_all
for code in sorted(ko_feed, key=lambda c: int(c[1:])):
    fa, fb = ko_feed[code]
    wa = res[fa][2] if fa in res else None
    wb = res[fb][2] if fb in res else None
    if wa and wb:
        orient(code, wa, wb)

Penalty shootouts are the edge case. The football-data feed folds the shootout goals into the full-time score, so the bot subtracts the penalty tally to recover the real level score.

wc26-bracket/scripts/fetch_results.py:89-143
if shootout:
    ph, pa = int(ph), int(pa)
    # football-data.org folds the shootout goals into fullTime (so a
    # 1-1 game won 4-3 on pens reports as 5-4). Recover the level
    # regulation/ET score by subtracting the shootout tally.
    if gh != ga:
        rh, ra = int(gh) - ph, int(ga) - pa

It is idempotent. A run with no new finished games writes no change, and it never overwrites a good result with junk.

The running example

Match M74 finishes 1 to 1, Paraguay winning 4 to 3 on penalties. Without the fix, the feed reports a 5 to 4 win and the bracket shows the wrong story. The bot subtracts the penalties, records Germany 1, Paraguay 1, winner Paraguay, note "4-3 pens", and your Paraguay pick reads as a hit.

The machine keeps itself current

Technical

A GitHub Action runs the sync three times a day, timed after each match block.

wc26-bracket/.github/workflows/sync-results.yml
schedule:
  - cron: "0 22 * * *"    # ~6:00 PM ET
  - cron: "30 1 * * *"    # ~9:30 PM ET
  - cron: "0 6 * * *"     # ~2:00 AM ET
permissions:
  contents: write
  actions: write

One trap hides here. A push made with the built-in token never fires other workflows, so the site would stop redeploying on its own. The sync dispatches the deploy itself, which is why it needs actions: write.

wc26-bracket/.github/workflows/sync-results.yml
run: gh workflow run deploy-pages.yml --ref "${{ github.ref_name }}"

A CI drift guard closes the loop. It regenerates the HTML and fails the build if the output differs from the committed file, so the generator and the page never drift apart.

wc26-bracket/.github/workflows/ci.yml
run: |
  python scripts/build_dashboard.py
  git diff --exit-code docs/index.html
Run it yourself python scripts/build_dashboard.py
python scripts/fetch_results.py --dry-run
python -m unittest discover -s tests -v
The tradeoff

Derive from data instead of editing HTML. A hand-edit drifts the moment the data changes. Derivation means the sync only ever writes one JSON file, and the page rebuilds itself. It retired an older approach. The old one rewrote the Python source with regular expressions.

Build the machine, not the webpage.

03
Make it theirs

Into the browser

my-wc26-bracket client-side JavaScript

Lead voices: Karpathy, then Willison. The pure function is the simplest honest idea, and a test proves the port by running it.

I shared the build kit, and nobody built one. The friction was the instructions themselves. So everything the Python generator did on my machine had to run in the visitor's browser instead. Same output, from their file, with nothing to install.

One pure function

ELI5

Give one function your picks, the scores, and the tree, and it hands back the whole scored board. Same inputs, same output, every time.

Technical

The port rests on computeState(picks, live, topology), a pure transform. Section builders are pure string functions over its result.

my-wc26-bracket/docs/js/render.js:60-65
export function computeState(picks, live, topology) {
  const D = {};
  D.ENTRANT = picks.entrant; D.TIEBREAKER = picks.tiebreaker;
  D.R32 = picks.r32.map(r => r.slice(0, 5));   // [code,date,a,b,pick]
  D.R16_WIN = picks.r16_win; D.QF_WIN = picks.qf_win; D.SF_WIN = picks.sf_win;
  // ...derives scores, KPIs, and reach status from here
}

A test to prove the port

ELI5

A big rewrite hides small bugs. So the test checks the new browser output against the original, letter for letter. Not looks the same. Identical.

Technical

The golden test renders each section with the JavaScript engine and compares it to the Python reference, section by section.

my-wc26-bracket/tests/golden.mjs:11-27
const D = R.computeState(picks, live, topo);
const js = {
  bracket_actual: R.buildBracket(D, "actual"),
  scorecard: R.buildScorecard(D), kpis: R.buildKpis(D),
};
for (const k of Object.keys(py)) {
  if (py[k] === js[k]) { console.log("  ok   " + k); continue; }
  fails++;
}

In hindsight it is an eval. A regression gate you run, not a claim you trust.

Storage without leaks between brackets

Technical

The parsed bracket lives in localStorage. What-if score edits are tied to a hash of the bracket, so two uploads on one device never bleed into each other.

my-wc26-bracket/docs/js/storage.js:4-6, 30-32
const KEY = "wcb.fan.picks.v1";
const KEY_SCORES = "wcb.scores.v3";   // manual what-if overrides
export function resetWhatIfsIfChanged(picks) {
  const h = hashPicks(picks);
  if (safeGet(KEY_HASH) !== h) { safeDel(KEY_SCORES); safeSet(KEY_HASH, h); }
}
The running example

Your workbook holds M74 with Paraguay picked. The parser reads it into picks.r32, computeState scores it against the synced result, and the scorecard marks M74 a hit. All of it in your browser, with your file never leaving the device.

Run it yourself cd docs && python3 -m http.server 8000
node tests/golden.mjs
The tradeoff

A pure function is a promise you test. Moving the engine into the browser risked two code paths and drift. Expressing it as one pure function let a golden test lock browser output to the original, so the port stayed honest.

Ship the product, not the instructions.

04
Make it social

The URL is the database

sled-mywcbracket zero-backend social

Lead voices: Naval, then Litt. The durable idea. The link carries the data, and the design stays legible and tweakable.

Brackets are a social game. Every obvious way to compare with friends meant a backend, and a central place holding coworkers' names. The question was whether a link carries the bracket itself, so adding someone to your board never touches a server.

The wire format

ELI5

A bracket is 31 win-or-lose choices, so 31 ones and zeros. Pack them into the end of a link. The end of a link, after the number sign, is never sent to any server. The privacy is not a promise. It is how links work.

Technical

Encoding walks a canonical code order and writes one bit per match, then wraps the bits as JSON and base64url. Later rounds depend on earlier picks, so encode and decode both walk the same order.

sled-mywcbracket/docs/js/share.js:7-26
function codeOrder(S) {
  return [...S.r32codes, ...S.r16codes, ...S.qfcodes, ...S.sfcodes, S.finalcode];
}
export function encodeShare(picks, topology) {
  const S = deriveStructure(topology);
  const sel = {};
  picks.r32.forEach(m => { sel[m[0]] = m[4]; });
  // ...r16, qf, sf, final
  let bits = "";  // one "0" or "1" per code, in codeOrder
}

Decoding reverses it, then rejects a truncated or newer-version link with a clear message rather than rendering garbage.

sled-mywcbracket/docs/js/share.js:36-45
if (payload.v !== 1) throw new Error("this link was made by a newer version of the site");
if (typeof payload.b !== "string" || !/^[01]{31}$/.test(payload.b))
  throw new Error("the link doesn't contain a complete bracket");
The running example

Your M74 pick reaches the end of the road. It is one bit in the string. Paraguay was the second team listed, so the bit reads 1. All 31 bits, your name, and the tiebreaker pack into about 140 characters after the #. You text the link. Nothing was written anywhere.

Technical

Rivals live in one browser key, capped at twenty. Adding a duplicate is a no-op.

Privacy by design, and hermetic tests

Technical

The spec names the invariants as non-negotiable. No network write in the social code. A bracket leaves the browser only inside a link its owner made. Rivals stay local. Viewing a shared or demo bracket never mutates your own saved bracket. There is no re-share feature, on purpose, because rebroadcasting a coworker's picks without consent is the leak the design avoids.

The tests are hermetic. The golden snapshot renders against frozen fixtures, not the live results file a bot rewrites all day.

The bracket reads the real path

ELI5

The "actual path" view shows who really advanced, read straight from results. When your pick busted, it shows a placeholder like "Winner M97" instead of a blank hole.

Technical

Later slots read the real feeder winner from results, not the user's pick tree, so a busted pick never leaves a dead cell.

The tradeoff

base64 of JSON, not bit-packing. Tighter packing saves a few characters and costs readability. base64 of JSON fits any chat and reads with atob(). The bigger call was no backend at all. A free tier is a bill or a shutdown later. A link plus localStorage has no such cliff.

If the data is the link, there is nothing to leak.

Engineering walls

Five failure modes, and the root cause of each

Lead voice: Russinovich. The interesting part is what broke underneath.

Root cause

GitHub Pages refused to deploy

Pages needs an explicit enable and a deploy from a workflow. The first pushes assumed it worked. The fix was an explicit deploy workflow with auto-enablement and a two-attempt retry.

Live is a feature you build, not a checkbox you tick.

wc26-bracket 27cece7 · my-wc26-bracket 965af7d

Root cause

The sync bot did not trigger the deploy

A push made with the built-in token never fires other workflows, by design. So the site stopped redeploying after each sync. The fix was to dispatch the deploy from the sync job, with the actions: write permission.

Automation has blast radius. Know what your bot triggers and what it does not.

wc26-bracket 7d24710 · 9bab6f6

Root cause

The test read live data

The golden test first compared against live results, so every finished game broke the build for no bug. Freezing the inputs into tests/fixtures/ made it hermetic. It fails only on a code change.

A test tied to the outside world is a pager crying wolf.

sled-mywcbracket 003c056

Root cause

Shared storage wiped edits

Viewing a demo or a shared link reused the same storage as your own bracket, so peeking at another board erased your edits. The fix stashes your state before showing a borrowed bracket, restores it after, and keeps viewing read-only.

The scariest bugs are silent. No surprise data loss, ever.

my-wc26-bracket 6f0d64d · sled-mywcbracket 97e3c67

Fixed

Eight blank cells in the bracket map

The actual-path view read from the pick tree, so busted picks left holes. The rebuild reads each slot from results, with a "Winner M97" placeholder for undecided games.

Derive from the source of truth, not a stale copy of it.

What to keep

Transferable principles

Lead voice: Naval. Strip away the World Cup, and these remain.

1 · The spec is the product

A precise brief is a reusable artifact. Plans and code regenerate from it. When code drifts, the spec is the source of truth you diff against.

2 · Derive, do not hardcode

Compute the output from the data. A hand-edit drifts the moment the data changes. Derivation keeps the whole thing in sync for free.

3 · A pure function is testable

Same inputs, same output, no side effects. It let a golden test lock a browser port to the original, byte for byte.

4 · Tests must be hermetic

Freeze the inputs. A test reading live data fails for reasons no bug caused, and you learn to ignore it.

5 · The URL holds the state

31 picks are 31 bits. Packed into a fragment, they never reach a server, so a share needs no backend and leaks nothing.

6 · Know what your automation pushes

Bots step on each other. A push with the default token does not trigger other workflows. Set permissions on purpose.

Honest limits

  • The $0 cost holds because the browser does the work and GitHub serves static files. Heavy write traffic would need a different design.
  • A link is the data, so it cannot be revoked. The mitigation is a share-as alias, not a delete button.
  • Rivals live on one device. Cross-device sync is out of scope, on purpose.

If I started again

  • Write the data contract before any runtime, not after the first one shipped.
  • Freeze the test inputs on day one, so the eval never flaps.
  • Set the deploy and automation permissions up front. Most walls were CI, not code.
  • Stand up the dev-to-prod split from the start, not once production hurt.

What carries to the next build

The next build might be an agent, a process automation, or another page. The same habits hold. Each one adapts to the shape of the work.

  • Brief first. Intent is the artifact. For an agent, the spec matters more, not less, because the model has more room to wander.
  • Evals before scale. The golden test becomes a grader. Build the eval before the agent, and let it gate every change.
  • Guardrails by structure. The URL fragment made a leak impossible. For an agent, the same idea is tight tool scopes and a sandbox, so a bad step never reaches production.
  • Dev, test, prod. A bench to pressure-test the loop against real inputs before it ships.
  • Judgment stays with me. The model drafts and executes. I set direction, read the output, and own the call.
