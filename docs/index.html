01
The spark

An Excel sheet in my inbox

Every big build starts as a small annoyance. A leader in my org ran a World Cup bracket challenge, the office version. He emailed an Excel workbook. Everyone filled in their picks, 31 knockout games, a champion, and a tiebreaker, and emailed it back. He tallied the scores as the tournament went on.

I filled mine out like everyone else. Then I stared at the spreadsheet and saw more. All the drama of a World Cup, the upsets, the busted picks, the my-champion-is-still-alive tension, flattened into cells. The data wanted to be visual.

Design before code

I opened Microsoft Cowork and designed what I wanted. A glassmorphism dashboard. KPI cards for points confirmed and points still open. And the BracketMap, the piece I was proudest of. The first plan was small. View it in Cowork and schedule a sync for new results.

The plan grew. Instead of keeping it as a prompt, I packaged the design as a markdown build kit. A spec you hand to an AI to get the same dashboard with your own picks. Version 6, then 7, then 8, each one tighter. I did not know the term yet. I had found the idea behind everything after. The spec is the product. The code is the output.

02
Make it mine

Crossing into GitHub

wc26-bracket 125 commits · Jul 2 to 7

Hosting on GitHub Pages had sat on my someday list for a long time. This was the reason to do it. A dashboard at a URL means you check it from anywhere. The data refreshes itself. And the whole build stays in commit history, open to anyone, including future me.

I opened GitHub Copilot and we started. One hardcoded HTML file came apart, on purpose, into a real architecture.

  • The data layer. Picks and live results moved into data/*.json, so content and presentation stopped tangling.
  • The generator. build_dashboard.py turns picks and results into the finished page.
  • The sync engine. fetch_results.py pulls finished games from FIFA's public feed.
  • The automation. A GitHub Action runs the sync three times a day, commits results, and redeploys.

125 commits in six days. The number is not a flex. It is what learning in public looks like. Every commit is me working out git, workflows, CI, and deployment by doing it, with the receipts saved.

I was not maintaining a webpage. I had built a small machine, and it keeps itself current.

03
Make it theirs

The wall: nobody follows instructions

my-wc26-bracket 19 commits · Jul 7

I was proud of the build kit. I shared it. Then I watched what happened. Nothing. The person I shared it with was never going to walk through repo setup, Python scripts, and deployment to get a dashboard. Of course not. A month earlier, neither would I.

It stung for an hour. Then it reframed the problem. The friction was not in my instructions. The friction was the instructions.

Ship the product, not the instructions.

Here is the new goal. You drop in your Excel bracket and your dashboard appears. Everything the Python generator did on my machine now had to run in your browser. Copilot and I ported the render engine to client-side JavaScript. It parses the workbook with SheetJS, validates it, scores it live, and draws the map. For people without a spreadsheet, we added a click-to-pick builder. The tool stopped being mine and became anyone's.

A port this size breeds subtle bugs. So we built a golden test. It compares the JavaScript output against the original Python output, byte for byte. Not looks the same. Identical. In hindsight it is an eval, a regression gate, run against frozen inputs so live results never make it flap.

04
Make it social

The big idea: social, with no backend

sled-mywcbracket 43 commits · Jul 7 to 8

Single dashboards worked. Brackets are a social game. The point is watching how you stack up against your friends. Every obvious build meant a backend. Accounts, a database, hosting bills, and the real dealbreaker, a central place holding my coworkers' real names whether they opted in or not.

I kept turning one idea over. A share link carries the bracket itself, so adding someone to your leaderboard never touches a server.

It fits in a link.

A bracket is 31 win-or-lose choices. 31 bits, plus a display name and a tiebreaker. Encoded, it runs about 140 characters inside the URL fragment, the part after the #. By HTTP's own rules, browsers never send the fragment to a server. Not to GitHub. Not to anyone. The privacy is not a policy. It is physics.

Under the hood The whole bracket, encoded into a URL ▾

The wire format stays boring and readable. base64url of a small JSON object. The careful part is the bit order. Later rounds depend on earlier picks, so encode and decode both walk the tree in the same order.

sled-mywcbracket/docs/js/share.js, encode
export function encodeShare(picks, topology) {
  const S = deriveStructure(topology);
  const sel = {};
  picks.r32.forEach(m => { sel[m[0]] = m[4]; });
  S.r16codes.forEach((c, j) => { sel[c] = picks.r16_win[j]; });
  // …qf, sf, final…
  let bits = "";
  for (const c of codeOrder(S)) {
    const [a, b] = teamsFor(S, c, sel);
    bits += sel[c] === a ? "0" : "1";
  }
  const payload = { v: 1, b: bits, n: String(picks.entrant || ""), t: picks.tiebreaker | 0 };
  return b64url(new TextEncoder().encode(JSON.stringify(payload)));
}

Decoding reverses it, then revalidates against the fixed tree. A truncated or newer-version link fails with a clear message instead of rendering garbage. Hanselman's rule. Every error says what broke.

Where the workflow leveled up

I stopped prompting. I started briefing.

By the third build I had read what people shipping with AI agents do. The pattern holds. They do not prompt. They brief. So before any code, I wrote one. Every brief since follows the same four parts.

G

Goal

The outcome, stated so sharply you know if you miss it.

C

Context

Constraints, history, who it serves, and what exists.

S

The ground truth to build from. Repos, data, prior art.

E

Expectations

Acceptance criteria. What done looks like, checked.

My gap was not syntax. It was taste. Knowing when something turns over-engineered, badly named, or built for the wrong horizon. So I wrote a council of nine named voices into the repo instructions. Each owns a domain. The AI channels them on every real call.

The calls I am proudest of are the ones I did not build

  • No backend, not even a small one. Free tiers turn into a bill or a shutdown later. Links and localStorage have no such cliff.
  • No re-share of someone else's bracket. Rebroadcasting a coworker's picks without consent is the leak the design avoids.
  • base64 of JSON, not bit-packing. About 140 characters fits any chat and reads with atob(). The extra cleverness did not earn its keep.
The walls

Every one taught me something

A build story with no walls is a brochure. Here are the five walls. Each one stopped me. What broke, how I got past it, and the commit to prove it.

Broke

GitHub Pages refused to deploy

The first pushes did not publish. Pages needs an explicit enable and a deploy from a workflow. I moved to an explicit deploy workflow with auto-enablement and a two-attempt retry, because it ran flaky.

Live is a feature you build, not a checkbox you tick.

wc26-bracket 27cece7 · deploy Pages from an explicit workflow · my-wc26-bracket 965af7d · enable Pages auto-enablement in the deploy workflow (#1)

Broke

My own sync bot pushed against me

The sync committed new results with the default token. By GitHub's design, those commits do not fire other workflows. So the site quietly stopped redeploying. The fix. The sync job dispatches the deploy itself, with the actions: write permission to do it.

Automation has blast radius. Know what your bot triggers and what it does not.

wc26-bracket 7d24710 · make auto-sync commits trigger a Pages deploy · 9bab6f6 · give the sync job actions:write to trigger the deploy

Broke

My tests would break every match day

The golden test first compared against live results. Every finished game would break the build for a reason no bug caused. I froze the test inputs into tests/fixtures/, so the eval stays hermetic. It fails only when the code changes.

A test tied to the outside world is not a test. It is a pager crying wolf.

sled-mywcbracket 003c056 · the hermetic golden snapshot and CI-tests stage

Broke

Viewing the demo silently deleted my edits

Opening the demo or a shared link reused the same storage as your own bracket. Peeking at another board wiped your what-if edits. The fix. Stash your state before showing a borrowed bracket, restore it after, and keep viewing read-only.

The scariest bugs are silent. No surprise data loss, ever.

my-wc26-bracket 6f0d64d · demo becomes preview-only · sled-mywcbracket 97e3c67 · share links and view-only mode

Fixed

Eight blank cells in the bracket map

The actual-path view read from your pick tree. Once your picks busted, real advancers fell off and left holes. I rebuilt it to read each slot from results. Real occupant if decided, a dashed “Winner M97” placeholder if not, independent of your picks.

Derive from the source of truth, not a stale copy of it.

The return

Back to where it started

The prize I came back with was not the dashboard. It was a way of working I repeat on anything. Set a vision. Understand the outcome. Write the brief. Shape it with judgment.

What happened

A leader in my org asked to share it broadly. Coworkers started opening each other's links and comparing boards. For a weekend fan project, this is the traction I want. People chose to use it and to pass it on.

What I would measure next

A leader asking to share it is a signal, not a metric. If I kept going, I would instrument the shares and the count of active leaderboards. Two numbers tell me whether the social loop compounds.

What I would do differently. Still learning.

  • Freeze the test inputs on day one, not after the first false alarm.
  • Decide the deploy and automation permissions up front. Most walls were CI, not code.
  • Write the SPEC before the first line, every time. It saved the most time on repo 3.

The codebase is public. If you see a better way, I want to hear it. Half the reason it is open.
