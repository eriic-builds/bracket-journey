Part 1 · Plans execution

From one brief to thirteen plans

The whole build ran off a single brief (kept in /input/brief.md) turned into a plan (/input/plan.md) of 13 tracked steps. They ran in five phases. Foundations first, then everything able to run in parallel, then assembly and shipping.

ELI5

Think of it like building a float for a parade. First you build the frame (foundations). Then a bunch of people work at the same time, one paints, one writes the signs, one takes photos. Finally everyone bolts their pieces on, you double-check it, and you roll it out.

Technical

Todos ran with explicit dependencies (a small DAG). Phase 1 (scaffold → design tokens → motion) ran serial because everything inherits from it. Phase 2 (git-history mining, Playwright screenshot capture, imagery) ran fully independent, as background agents and async jobs. Phase 3 (narrative, architecture panels, skim layer) depended only on the mined facts. Phase 4 assembled and hardened. Phase 5 shipped to Pages.

Part 2 · Codebase analysis

What's in the repo, and how it fits

The whole thing stays deliberately boring in the best way. No framework, no build step, no dependencies. A browser opens docs/index.html directly. It matches the discipline of the apps it documents.

ELI5

It is one web page, one stylesheet, and one small script, like a poster, a box of crayons, and a wind-up motor. Nothing to install. Double-click and it runs.

Technical

A static site served from docs/ on GitHub Pages. journey.css holds HSL-free design tokens per [data-theme]. motion.js is an IIFE using IntersectionObserver and one rAF-batched scroll handler. Progressive enhancement. With JS off, content still renders and reveals show right away.

ELI5

Three little machines. One turns your spreadsheet into a scoreboard, one fetches the real scores a few times a day, and one lets you mail your scoreboard to a friend inside a link, with no post office (server) in the middle.

Technical

Repo 1 generates statically (Python → HTML) and syncs via Actions. Repo 2 makes the render engine a pure function and runs it client-side (SheetJS parse plus golden-test parity). Repo 3 serializes the 31-bit pick vector to base64url(JSON) in the URL fragment, never transmitted, and keeps rivals in localStorage (wcb.rivals.v1).

Part 4 · What the review changed

The revisions I made because of it

A review changing nothing is theater. These landed before ship, straight from the voices above.

Accessibility

Skip-to-content link, visible focus rings, honest alt text, and a graceful placeholder for any image failing to load. Hanselman

Performance

Screenshots optimized about 7× smaller, lazy-loaded and async-decoded. Scroll work batched in one rAF. Motion off under reduced-motion. Russinovich

Copyright and leverage

No stock photos or FIFA, MS, or GitHub marks. Only real screenshots and CSS. Less risk, less weight, more honest. Naval

Truthfulness

Verbatim-only code, real commit hashes, computed metrics, and a 0-error headless render check. Willison

Spec in the repo

Brief and plan committed to /input and linked, so intent survives the code. Grove

Part 5 · Learning moments

The transferable lessons

Strip away the World Cup and these are the ideas worth keeping, each one ELI5 and technical, because you asked to learn both.

1 · The spec is the product

ELI5

Write down exactly what "good" looks like first. The instructions are the real thing you make. The code is what pops out.

Technical

A precise brief (Goal, Context, Source, Expectations) is a reusable, model-agnostic artifact. Plans and code regenerate from it. When code drifts, the spec is the source of truth you diff against.

2 · The URL is the database

ELI5

A link carries the whole thing inside it. If the secret part never reaches any computer but your friend's, nothing leaks.

Technical

31 picks = 31 bits → base64url(JSON) in the URL fragment. By spec, fragments never go in HTTP requests, so a share is zero-backend and privacy is structural, not policy.

3 · Tests must be hermetic

ELI5

Do not grade your homework against a scoreboard changing every day, or you fail every time a real game ends. Freeze the answer key.

Technical

The golden test first compared against live results and flapped on every match. Freezing inputs to tests/fixtures/ made it a deterministic regression eval, failing only on a code change.

4 · Derive from the source of truth

ELI5

Show what happened, not a copy of your guess. When a guess was wrong, the copy leaves holes.

Technical

The actual-path bracket read from the user's pick tree, so busted picks left 8 blank slots. Rebuilt to read each slot from results, with placeholders for undecided games.

5 · Automation has blast radius

ELI5

Robots you build step on each other by mistake. Know exactly what each one pushes.

Technical

Commits made with the default GITHUB_TOKEN never trigger other workflows, so the site stopped redeploying. The fix. An explicit gh workflow run dispatch plus actions: write.

6 · Cheap models type, expensive models think

ELI5

Have your smartest helper plan and check the work. Let faster helpers do the typing in between. You stay the boss who approves.

Technical

Brief → strong model plans → human review → cheaper models execute → strong model reviews → human approves. The expensive model reasons twice, plan and grade. Throughput comes from the cheaper executors.

If you remember one thing

Set the vision, write the brief, and keep judgment in the loop. The tools change. The loop is the durable skill, and it turned a spreadsheet into three live apps in a weekend.
